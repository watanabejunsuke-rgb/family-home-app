# 通知の実装計画

暮らしnoteの通知を、段階的に「アプリ内お知らせ → LINEプッシュ通知」まで育てるための計画。

## 現状(v0.12〜)

- **アプリ内お知らせ(実装済み)**: ホーム右上のベルに、今日/期限まわりの項目を集約表示。件数バッジ付き。
  - 元データは `App.data.notifications()`(`js/store.js`)が **localStorage の実データから毎回導出**。完了・水やり等で対応すると自然に消える。
  - 種類(予定/やること/植物/試合)のオン・オフは メニュー →「通知・お知らせ」(`settings.notifPrefs`)で制御。
- **まだ無いもの**: アプリを開いていないときに届く **LINEプッシュ通知**。

## なぜ今のSPA単体では届かないか

- LIFF(ミニアプリ)は**開いている間だけ**動く。バックグラウンド配信はできない。
- LINEへ能動的に送るには **Messaging API の push/multicast** が必要で、これは**サーバー(実行主体)**からしか呼べない。
- データが**端末内(localStorage)**にあるので、サーバーは「誰に何を送るか」を知らない。→ 送信の前提として**共有ストレージへの同期**が要る。

## 構成案(GAS最小構成)

非エンジニアでも回しやすいよう、Google Apps Script + スプレッドシートをバックエンドにする。

```
[LIFFアプリ] --(1) userId+データをPOST--> [GAS doPost] --> [スプレッドシート(DB)]
                                                  ^
[GAS 時間主導トリガー(毎朝7時)] --(2) その日の通知を算出--> [LINE Messaging API push]
```

必要なもの:
- LINE公式アカウント(Messaging APIチャネル)＋**チャネルアクセストークン**
- LIFFで `liff.getProfile()` から取得する **userId**(送信の宛先)
- ユーザーが公式アカウントを**友だち追加**していること(push送信の条件)

## 段階(小さく出す順)

1. ~~**userIdの取得と保存**~~ → **完了(2026-07-15、世帯共有機能の一部として実装)**。LIFFのIDトークンをGAS側で検証し、`sub`(userId)を`households`シートの`members`に記録。
2. ~~**データのサーバー同期**~~ → **完了(2026-07-15)**。世帯共有の`push`/`pull`で予定・やること・買い物・植物・メモがGASスプレッドシートに同期される。
3. ~~**毎朝のダイジェスト push(1通)**~~ → **運用開始(2026-07-15)**。`backend/Code.gs`の`sendDailyDigest()`が`js/store.js`の`notifications()`と同一ロジックをGAS側に移植し、世帯ごとに今日のやること/予定/植物/試合をまとめて1通pushする。LINE公式アカウント作成・友だち追加・トークン発行・毎朝6時台のトリガー登録まで完了し、実際にLINEへの着信を確認済み。
   - **予定・試合の「誰の予定か」**(2026-07-15追加): 家族全員が対象(「みんな」)のときは何も付けず、一部のメンバーだけが対象の時だけ`whoSuffix()`が名前を「・」区切りで末尾に追記する(例:「こた めばえ(今日16:30・こた)」)。「みんな」の予定にまで機械的に付けるとノイズになるとの判断で、この絞り込みを採用。
   - **アプリを開くリンクを添付**(2026-07-15追加): メッセージ末尾に常時`LIFF_URL`を付け、通知からそのままアプリを開けるようにした。
   - **通知設定(notifPrefs)の個人ごとのサーバー同期**(2026-07-15完了): 世帯の共有dataとは別枠で、`households`シートに列F`memberPrefs`(`{userId: notifPrefs}`)を新設。`setNotifPrefs`アクションでuserIdごとに保存し、`sendDailyDigest`は**受信者ごとに**自分の設定を反映した文面を組み立てて送る(世帯共通の1通ではなくなった)。フロントは`App.sync.pushNotifPrefs()`が`notifSettings`画面のトグル操作・世帯作成/参加のたびにサーバーへ送信。既存データに列Fが無くてもparseJsonが`{}`にフォールバックするため、未設定時は「全部オン」の従来動作のまま安全に動く。
4. **個別リマインド**: 予定の30分前など、きめ細かい通知は最後。トリガー粒度と無料枠(送信数)を見てから。

## リッチメッセージ化(Flex Message)+双方向操作(postback)【実装済み・2026-07-16】

計画段階では「優先度低め・次のフェーズで」としていたが、ユーザーからの要望を受けて先行実装した。

- **Flex Message化**: `buildDigestText`(プレーンテキスト版・参照用に残置)とは別に`buildDigestFlex`を新設。
  やること・植物のお世話には「完了」ボタン付きの行を、予定・試合には情報のみの行を、天気ひとことは
  ヘッダー直下に表示するカード形式に組み直した。`sendDailyDigest`は今後こちらを使う
- **双方向操作(postback)**: LINEのFlex Messageの「完了」ボタンを押すと、`doPost`が
  (アプリ自身のAPI呼び出しとは別の)LINE Webhookイベントとして検知し、該当のタスク完了・植物の水やり/
  お世話完了をスプレッドシート上のデータに直接反映する。反映後はreplyTokenで「「〇〇」を完了にしました」
  と返信する
- **セキュリティ上の制約と対処**: GASの`doPost(e)`は**HTTPヘッダーを読めない**仕様のため、LINEの標準的な
  署名検証(`X-Line-Signature`ヘッダーを使う方式)が使えない。代わりに、Webhook URLの末尾に
  `?webhookToken=(秘密の文字列)`を付与し、スクリプトプロパティ`WEBHOOK_TOKEN`と一致するかで正当性を
  確認する方式にした。この文字列は公開リポジトリのフロント設定には一切含まれず、LINE Developers
  コンソールのWebhook URL欄にのみ設定する
- **セットアップに必要な作業(ユーザー側)**:
  1. スクリプトプロパティに`WEBHOOK_TOKEN`(適当な長い英数字)を追加
  2. LINE Developersコンソール → Messaging API設定 → Webhook URLに
     `(GASのWebアプリURL)?webhookToken=(1で決めた文字列)`を設定し、「Webhookの利用」をオン
  3. 応答メッセージ・あいさつメッセージ等の自動応答機能は、Webhookの「完了」ボタン以外への反応と
     競合しないよう、LINE Official Account Manager側でオフにしておくことを推奨
  4. `backend/Code.gs`を最新化してデプロイ(新しいバージョン)
- **ハマりどころ(2026-07-16発生・解決済み)**: 「Webhook URL」に、GASの**Web アプリのURL**
  (`https://script.google.com/macros/s/(長いID)/exec`)ではなく、**GASエディタの「プロジェクトの設定」画面
  のURL**(`https://script.google.com/u/1/home/projects/(ID)/settings`、ブラウザのアドレスバーから
  誤ってコピーしがち)を登録してしまい、ボタンを押しても何も起きない(botからの返信が来ない)という
  不具合が起きた。この誤ったURLはGoogleログインが必須のページのため、LINE側のアクセスは302で
  弾かれ続け、`doPost`にすら届いていなかった。厄介なのは、実行ログ(実行数)に`doPost`の実行が
  残っていたこと──これはWebhookの成功ではなく、**アプリ自身のバックグラウンド同期(pull/push)**が
  同じ`doPost`を呼んでいたのが操作タイミングとたまたま重なっていただけだった。正しいURLは
  「デプロイを管理」→ 該当デプロイ →「ウェブアプリ」欄に表示される`.../exec`形式のもの。
  次回似た設定をする際は、コピー元を「ブラウザのアドレスバー」ではなく必ず「デプロイ管理画面の
  ウェブアプリURL欄」にすること。

## 今後の余地(計画のみ・未着手)

- 双方向操作の対象を「完了」以外の操作(例:やることの延期、植物の「今年は見送り」)にも広げる
- Flex Messageのデザイン調整(ボタン配置・色味など、実際にLINEで届いた見た目を見てから微調整)

## 注意・制約

- 無料枠の送信数上限(プランにより月数百通〜)。**1日1通のダイジェスト**なら家族数人でも十分収まる。
- 送信内容に住所・電話等の個人情報を載せない(メッセージは家庭内向けでも最小限に)。
- トークンはGAS側のスクリプトプロパティに保存し、フロント(公開リポジトリ)には**絶対に置かない**。

## このリポジトリ側で先にできる準備

- `notifications()` を**種類判定込みで実装済み**なので、サーバー側は同じロジックを移植するだけでよい(算出仕様が二重にならないよう、仕様はこの関数を正とする)。
- `settings.notifPrefs` のキー(`event` / `task` / `plant` / `match`)を**サーバーと共通の契約**として扱う。
