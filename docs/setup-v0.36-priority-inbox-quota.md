# v0.36.0 セットアップ・移行・テスト・ロールバック手順

対象: ホーム「まず確認」/ 予定アクションパック(準備)/ LINE家族インボックス / 月200通クォータ / 匿名の利用計測

**設計の大原則**: すべての新機能はフラグ制御で、フラグOFF(既定)なら従来どおり動きます。
フロントは `js/config.js` の `FEATURE_FLAGS`、GASはスクリプトプロパティの `FLAG_*` です。
両方は独立しているので、片方だけ有効にしても壊れません。

### データ構造(households.data のJSON。すべて後方互換)

- `event.preparation = { assignedTo, status, items: [{id, label, checked, confirmedBy, confirmedAt, sortOrder}] }`(任意)
- `event.important = true`(任意)/ `event.updatedAt = ms`(編集時に付く)
- `task.eventId` / `task.eventActionType: "preparation"` / `task.assignedTo`(任意)
- `data.inboxItems = [{id, text, from, receivedAt, status, processedAt, targetType, targetId}]`
  - status: `pending`(候補待ち) / `later`(あとで整理) / `processed`(登録済み)。最大200件で古い順に自動削除
- `data.notificationCenter = [{id, type, title, meta, createdAt}]`(GASが抑制時に書く。最大50件)

既存データにこれらのフィールドが無くても、全画面がガード済みで正常に動きます。
**既存データの一括変換・移行処理はありません(不要)**。

### スプレッドシート(GASの `setupNewFeatures()` が作成)

| シート | 用途 |

| `line_message_log` | 送信・抑制・エラーの記録。`dedupe_key` で再送防止 |

| `product_events` | 匿名の利用計測(本文は一切保存しない) |

`households` / `consultations` シートには**列の追加も変更もありません**。

## 2. セットアップ手順(渡辺さんの操作)

### 2-1. フロント(GitHub Pages)

`main` にpushするだけで反映されます(バージョンは `0.36.0` に更新済み。キャッシュクエリも全ファイル更新済み)。
フロントのフラグは `js/config.js` の `FEATURE_FLAGS` で個別にON/OFFできます。
初期値は `PRODUCT_ANALYTICS` だけ `false`(GAS側の準備が済んでから `true` に)。

### 2-2. GAS

1. `backend/Code.gs` の中身を丸ごとGASエディタに貼り付けて保存
2. 関数選択で **`setupNewFeatures`** を選んで実行(初回は権限承認が出たら許可)
   - 実行ログに「setupNewFeatures 完了」と出ればOK。何度実行しても安全
3. スクリプトプロパティに追加(**有効にしたい機能の分だけ**):
   - `FLAG_LINE_MESSAGE_QUOTA` = `true` ← まずこれを推奨(送信の安全装置)
   - `FLAG_LINE_INBOX` = `true`
   - `FLAG_PRODUCT_ANALYTICS` = `true`(任意)
4. **デプロイ → デプロイを管理 → 編集(鉛筆)→ バージョン: 新バージョン → デプロイ**
   - これを忘れると公開URLは古いコードのまま動きます(過去にハマった箇所)

### 2-3. LINE Developers(変更はほぼ不要)

- Webhook URLは従来のまま: `(GASのURL)?webhookToken=(WEBHOOK_TOKENの値)`
- 「Webhookの利用」がONであること
- 「応答メッセージ」はオフのまま(インボックスの返信はGASが行う)
- テキストを受信するために追加で必要な設定はありません

### 2-4. 推奨の有効化順序

1. GAS再デプロイ+`setupNewFeatures`(この時点では挙動不変)
2. `FLAG_LINE_MESSAGE_QUOTA=true` → 朝ダイジェストが翌朝も届くこと・`line_message_log`に記録されることを確認
3. `FLAG_LINE_INBOX=true` → 下記の手動確認
4. `FLAG_PRODUCT_ANALYTICS=true` と `js/config.js` の `PRODUCT_ANALYTICS: true`

## 4. 実LINEアカウント2つでの手動確認手順

前提: 渡辺さん(世帯作成者=管理者)と奥さんの2アカウント。両方が公式アカウントを友だち追加済み。

### A. インボックス(FLAG_LINE_INBOX=true 後)

1. 【渡辺】公式アカウントに「牛乳」と送る → 返信は**1通だけ**(「どこに登録しますか?」+ボタン: 買い物にする/やることにする/メモにする/あとで整理)
2. 「買い物にする」をタップ → 「登録しました」の返信1通 → ミニアプリの買い物リストに「牛乳」がある
3. **同じボタンをもう一度タップ** → 「すでに登録済みです」で、買い物リストに2件目ができていないこと(冪等性)
4. 【渡辺】「買い物:トマト」と送る → 返信なし(既定)で買い物リストに直接入る
   - 通知設定「LINE登録時の短い返信」をONにしてから送ると、確認の返信が1通届く
5. 【渡辺】「10月12日 9時 運動会」と送る → 候補に「予定にする」→ タップ → カレンダー10/12 9:00に登録
6. 【奥さん】何か送る → 同様に動き、登録結果が渡辺さんの端末にも同期される
7. 【第三者(世帯外)のアカウント】から送る → 何も起きない(登録も返信もされない)
8. ミニアプリ: メニュー →「未整理」→「あとで整理」にしたものが並び、そこから予定化できる

### B. クォータ(FLAG_LINE_MESSAGE_QUOTA=true 後)

1. GASエディタで `sendDailyDigest` を手動実行 → 両アカウントにダイジェストが届く
2. `line_message_log` に status=sent の行、`message_quota` の sent_count が増えている
3. **同じ日にもう一度 `sendDailyDigest` を実行** → 誰にも届かない(dedupe_keyで再送防止。ログにも新しいsentが増えない)
4. `message_quota` の `sent_count` を手で `195` に書き換え → 翌朝(または手動実行)のダイジェストは届かず、ミニアプリのベル(お知らせ)に「今朝のダイジェストはアプリでどうぞ」が出る。確認後は `sent_count` を元の値に戻す
5. sent_countを195のままインボックスに「牛乳」を送る → **返信は届く**(replyはcritical扱い)
6. ミニアプリ(渡辺=管理者): メニュー → 通知・お知らせ → 一番下「LINE通知の利用状況」→ 今月の送信数・残り・用途別が見える
7. 奥さんの端末では同じ画面に利用状況セクションが**出ない**こと

### C. 既存機能の回帰

1. 朝ダイジェストのカードから「完了」ボタン(タスク・水やり)→ 従来どおり完了+返信
2. 予定の追加・編集・くり返し・削除、写真の登録(植物)、世帯共有(招待コード)が従来どおり
3. 予定・やること・買い物・メモに何も無い日はダイジェストが届かない(天気だけでは送られない)

## 5. ロールバック方法

段階的に戻せます。**データは消えないので、どの段階でも安全です。**

| 戻したいもの | 操作 |
|---|---|
| インボックスだけ止める | スクリプトプロパティ `FLAG_LINE_INBOX` を `false`(または削除)。受信を無視する従来動作に戻る |
| クォータ管理だけ止める | `FLAG_LINE_MESSAGE_QUOTA` を `false`。送信は従来の直接方式に戻る(ログ・抑制なし) |
| 計測だけ止める | `FLAG_PRODUCT_ANALYTICS` を `false` と `js/config.js` の `PRODUCT_ANALYTICS: false` |
| フロントのUIだけ戻す | `js/config.js` の `FEATURE_FLAGS` を全て `false` にしてpush(VERSIONを上げて`?v=`一括更新を忘れずに) |
| フロントを丸ごと戻す | `git revert` でv0.36.0のコミットを打ち消してpush |
| GASを丸ごと戻す | デプロイを管理 → 以前のバージョンを選んでデプロイし直す(またはgitの旧Code.gsを貼り直し) |
| 追加シート | 消さなくてよい(フラグOFFなら読み書きされない)。完全に消す場合も `households` / `consultations` には影響しない |

注意: 旧Code.gsに戻すと、フロントが `inboxItems` / `notificationCenter` を共有データとしてpushしても、それ自体は `data` JSONの一部としてそのまま保存されるため問題ありません(旧GASは中身を解釈しないだけ)。

## 6. 運用メモ・既知の割り切り

- **返信のクォータ計上**は既定で「数える」(安全側)。LINE側の仕様でreplyが無料と確定していれば `QUOTA_COUNT_REPLIES=false` で切り替え
- 用途別の予算列(digest_budget等)は**記録・可視化用**で、送信可否はしきい値(160/190/200)で判定する
- 朝ダイジェストの「まず確認」専用枠は重複を避けるため入れていない(既存セクションが同じ対象をカバーするため)。全体で最大5項目+「ほか◯件」
- 担当者ごとのダイジェスト出し分けは「担当:名前」の表示まで。LINEアカウントと家族メンバーの対応表が無いため、本人分だけに絞る機能は次段階(family[].lineUserId の対応付けが前提)
- くり返し予定に付けた準備(持ち物・準備タスク)は**初回の1件だけ**に付く(毎回分に複製すると持ち物確認が氾濫するため)
- インボックスの保存は1件500文字まで・1世帯200件まで(古い順に自動削除)
- 見送り(まず確認)は端末ローカル・本人単位。家族には共有されない
